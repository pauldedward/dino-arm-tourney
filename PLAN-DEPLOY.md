## Constraints

| Goal | Choice |
|---|---|
| **Zero monthly cost** | Vercel Hobby + Supabase Free + Cloudflare R2 Free. No paid tier on any service. |
| **One-command rollouts** | `git push origin main` → Vercel auto-builds & promotes. PRs get preview URLs free. |
| **No vendor lock-in panic** | Next.js standalone build also runs on Cloudflare Pages, Fly.io free shared-cpu, or self-host (§ Fallback). |
| **Match-day safety** | Production deploys frozen `T-2 days` before any event. Hot-fixes go through preview URL → manual promote. |

## 0. Pre-flight (one-time, before first deploy)

- [ ] Decide production domain. Options:
  - `dino-arm-tourney.vercel.app` (free, instant).
  - `<custom>.com` via Cloudflare Registrar at cost (≈ ₹800/yr) + free Vercel custom-domain mapping. **Defer until users ask.**
- [ ] **Create a NEW Supabase project `dino-prod`** (region: ap-south-1 / Mumbai). Do **not** point production at the dev project `hmvnelyzqqdfidjalsha`. Reason: dev DB is wiped/reseeded freely (`npm run seed:reset`, `npm run users:reset`) — must never touch real users. See [DEPLOY-GUIDE.md](DEPLOY-GUIDE.md) Part B.
- [ ] **Create NEW R2 buckets `tm-prod-public` + `tm-prod-private`** (region: APAC). Existing buckets `tournament-manager` / `tournament-manager-media` stay as the dev pair. See [DEPLOY-GUIDE.md](DEPLOY-GUIDE.md) Part C.
- [ ] Generate a fresh **Vercel deploy hook** (Project → Settings → Git → Deploy Hooks) for manual re-deploy without a commit.

### Why dev ≠ prod (non-negotiable)

Until now the dev laptop and any future production deploy would have
shared one Supabase project + one R2 bucket pair. That is unsafe:

- Dev seeders (`web/scripts/seed-bulk.ts`, `populate-event.ts`,
  `drop-all-users.mjs`) freely create/delete users and events. They
  must never run against real federation data.
- Migrations under test on the dev DB shouldn't lock real-event tables.
- R2 dev uploads (test posters, junk PDFs from rehearsals) shouldn't
  inflate the prod 10 GB free quota.

Split gives full isolation at ₹0 (Supabase Free allows 2 projects per
org; R2 free tier is per-account, not per-bucket).

## 1. Vercel project setup (one-time, ~10 min)

1. https://vercel.com/new → **Import Git Repository** → select this repo.
2. **Root Directory**: `web` (not the repo root — the Next app lives here).
3. **Framework Preset**: Next.js (auto-detected).
4. **Build & Output**: leave defaults. `npm run build` works as-is.
5. **Environment Variables** — paste values from the `prod-secrets.txt`
   you built in [DEPLOY-GUIDE.md](DEPLOY-GUIDE.md) Parts B + C (the
   `dino-prod` Supabase + `tm-prod-*` R2 credentials, **not** the dev
   ones from `web/.env.local`). Mark each as
   **Production + Preview + Development**:

   | Var | Source | Sensitive |
   |---|---|---|
   | `NEXT_PUBLIC_SUPABASE_URL` | Supabase project settings | no |
   | `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Supabase project settings | no |
   | `SUPABASE_SERVICE_ROLE_KEY` | Supabase project settings | **yes** |
   | `R2_ACCOUNT_ID` | Cloudflare R2 dashboard | no |
   | `R2_ACCESS_KEY_ID` | R2 API token | **yes** |
   | `R2_SECRET_ACCESS_KEY` | R2 API token | **yes** |
   | `R2_PUBLIC_BUCKET` | `tournament-manager` | no |
   | `R2_PRIVATE_BUCKET` | `tournament-manager-media` | no |
   | `R2_PUBLIC_BASE_URL` | `https://pub-<hash>.r2.dev` (or custom) | no |
   | `NEXT_PUBLIC_APP_URL` | `https://<project>.vercel.app` | no |
   | `NEXT_PUBLIC_APP_ENV` | `production` | no |
   | `NEXT_PUBLIC_ENABLE_SW` | `1` | no |
   | `RAZORPAY_KEY_ID` / `_SECRET` | only if/when wired | **yes** |

6. **Production Branch**: `main`. **Preview**: every other branch + every PR.
7. Click **Deploy**. First build cold takes ~3 min (sharp + @react-pdf are heavy).

## 2. Supabase wiring

- [ ] **Site URL** (Auth → URL Configuration) → `https://<project>.vercel.app`.
- [ ] **Additional Redirect URLs** → add `https://*-<account>.vercel.app/**` so preview deploys can sign in too.
- [ ] **CORS** → no action (Supabase JS uses fetch with auth headers, not CORS-restricted).
- [ ] **First-day schema**: paste [supabase/schema.sql](supabase/schema.sql) into the prod SQL Editor (one shot, ~10s, includes the seeded `rule_profiles` rows). Bundle is generated by `npm run schema:bundle` which concatenates every file in `supabase/migrations/` in order — re-run it whenever a new migration lands so the bundle stays current for the *next* fresh setup.
- [ ] **First-day seed**: paste [supabase/seed.sql](supabase/seed.sql) into the prod SQL Editor (1 row — the default `tnawa` organisation, required because the app has no UI to create orgs and the New-Event form's org dropdown would be empty without it).
- [ ] **First super-admin**: open `https://<prod>/register-super-admin` (already shipped in-app, hard-coded to the owner email, auto-disables once a super-admin exists). After this, invite operators in-app at `/admin/users`.
- [ ] **Ongoing migrations**: continue committing new files under `supabase/migrations/`. Apply each new file manually via SQL Editor (or `supabase db push` from local CLI) **before** deploying any code that depends on it. Vercel does **not** run migrations — that's intentional (matches §5 Safety in `.github/copilot-instructions.md`).
  - Pre-deploy checklist: "Did this PR add a `supabase/migrations/*.sql`? If yes, has it been applied to the prod project? Has `schema.sql` been regenerated and committed?"

## 3. Cloudflare R2 wiring

- [ ] Public bucket `tournament-manager`: confirm **Public Access** is on; copy the `pub-<hash>.r2.dev` URL into `R2_PUBLIC_BASE_URL`.
- [ ] CORS rule on **public** bucket (R2 → Settings → CORS):
  ```json
  [{"AllowedOrigins":["https://<project>.vercel.app","http://localhost:3000"],
    "AllowedMethods":["GET","HEAD"],
    "AllowedHeaders":["*"],
    "MaxAgeSeconds":3600}]
  ```
- [ ] Private bucket `tournament-manager-media`: stays private. Server signs URLs (`@aws-sdk/s3-request-presigner`). No CORS needed — signed URLs are fetched server-side or as `<img>` (CORS only matters for fetch with credentials).
- [ ] R2 token scope: **Object Read & Write** on the two buckets only. Rotate every 6 months.

## 4. Rollout workflow (every change after first deploy)

```
local change
   │
   ├── tdd: red → green → refactor (web/src/**/*.test.ts)
   │
   ├── git commit -am "..."   ─►   git push origin <branch>
   │                                   │
   │                                   ▼
   │                             Vercel preview URL (auto)
   │                             https://<branch>-<hash>.vercel.app
   │
   ├── smoke-test preview URL (login, create event, register, weigh-in)
   │
   ├── if migration: run it on prod Supabase NOW (manual, see §2)
   │
   └── git checkout main && git merge --ff-only <branch> && git push
                                   │
                                   ▼
                             Vercel production deploy (~2 min)
                             https://<project>.vercel.app
```

**Rollback**: Vercel → Deployments → previous green build → "Promote to Production". One click, ~10s. Migrations are forward-only — if a deploy depended on a bad migration, write a follow-up migration that fixes/reverses it. Never `DROP` columns the previous build still reads.

## 5. Match-day freeze (mandatory)

| When | Action |
|---|---|
| T-2 days | Tag the current prod commit `event-<slug>-frozen`. No merges to `main` until event ends. |
| T-1 day | Run dress rehearsal against the prod URL (see [docs/dress-rehearsal.md](docs/dress-rehearsal.md)). Pre-cache photos to venue laptop (`npm run cache:photos`). |
| Event day | Hot-fixes go to a `hotfix/<slug>` branch → preview URL → smoke-test → manual promote. Never push directly to `main` from a phone at the venue. |
| T+1 day | Untag, resume normal rollouts. |

## 6. Cost ceiling alarms

Set calendar reminders to glance at these monthly. All free tiers degrade silently.

| Service | Free cap | Where to check | Alarm threshold |
|---|---|---|---|
| Vercel Hobby | 100 GB bandwidth/mo, 100 GB-hr fn execution | vercel.com → Project → Usage | 70% |
| Supabase | 500 MB DB, 5 GB egress, 50k MAU | supabase.com → Project → Reports | 70% on any axis |
| R2 | 10 GB storage, 1M class-A ops, 10M class-B ops | dash.cloudflare.com → R2 → Metrics | 60% storage |

If any alarm trips: re-read [PLAN-WEEK1.md](PLAN-WEEK1.md) §1.4 budget assumptions, identify which row blew up, fix at the source (compression, caching, or moving to R2). Do **not** upgrade to a paid tier without asking the owner first — this plan ships on ₹0/month.

## 7. Fallback (if Vercel ever throttles or goes down match-day)

Already documented in [PLAN-WEEK1.md](PLAN-WEEK1.md) §1.6: run `npm run build && npm run start` on the venue laptop, expose via **Cloudflare Tunnel** (free, valid TLS):

```powershell
# one-time, on venue laptop
winget install --id Cloudflare.cloudflared
cloudflared tunnel login
cloudflared tunnel create dino-venue
cloudflared tunnel route dns dino-venue dino-venue.<your-cf-domain>
# match day
cd web; npm run build; npm run start    # terminal 1
cloudflared tunnel run dino-venue       # terminal 2
```

Same env vars (`web/.env.local` works). Set `NEXT_PUBLIC_APP_URL` to the tunnel URL. Add the tunnel URL to Supabase Auth → Redirect URLs before relying on it. Tested in [docs/match-day.md](docs/match-day.md) — keep it warm; don't first-time-test it during an event.
