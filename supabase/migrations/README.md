## What goes where

| Folder | Meaning | Run on fresh prod? |
|---|---|---|
| `legacy/` | History. Applied to prod long ago. Already represented in `../schema.sql` + `../seed.sql`. | **No** — `schema.sql` already contains everything in `legacy/`. Pasting `legacy/*.sql` again on a fresh DB would duplicate-fail (idempotent guards mostly cover it, but skip anyway). |
| `migrations/` (root, excluding `legacy/`) | **Pending work.** Each file is a forward-only change waiting to be applied to prod. | Yes — apply each in numeric order **after** `schema.sql` + `seed.sql`. |

## A new prod DB always = `schema.sql` + `seed.sql` + every pending migration

That's the contract. `schema.sql` is regenerated by `npm run schema:bundle` from the
contents of `legacy/` only. So the moment you move a migration into `legacy/`, it
becomes part of the next bundle.

## Workflow for a new migration (`0045_*.sql` and beyond)

1. **Author** the file at `supabase/migrations/0045_<topic>.sql`.
   - Additive + idempotent: `create … if not exists`, `add column if not exists`,
     no destructive `drop column …` of anything the previous deploy still reads.
   - Two-phase for breaking changes (PR A adds new shape + dual-write; PR B backfills
     and drops old shape in a later deploy).
2. **Apply to dev** while iterating: `npm run dev` will fail loudly until the dev DB
   has it. Apply via Supabase SQL Editor (dev project) or `supabase db push`.
3. **Apply to prod** *before* merging the PR:
   - SQL Editor on `dino-prod`, **or**
   - `node scripts/apply-migrations.mjs --target prod --file 0045_x.sql --apply`
     (needs `SUPABASE_DB_URL` env pointing at prod).
4. **In the same PR, `git mv` the file into `legacy/`**:
   ```powershell
   git mv supabase/migrations/0045_<topic>.sql supabase/migrations/legacy/0045_<topic>.sql
   ```
5. **Re-bundle** so `schema.sql` includes the new file:
   ```powershell
   cd web; npm run schema:bundle
   git add ../supabase/schema.sql
   ```
6. Commit + push + open PR. CI must be green.
7. Merge to `main`. Vercel auto-deploys.

If you ship a PR where the migration is still in the root folder (not yet moved to
`legacy/`), it means the apply-to-prod step hasn't happened yet — **do not merge.**

## How to spot what still needs applying

Anything in `supabase/migrations/` that is **not** under `legacy/` is pending.
A clean repo at any point in time has zero pending files.

```powershell
Get-ChildItem supabase/migrations -File   # files here = not yet applied
```
